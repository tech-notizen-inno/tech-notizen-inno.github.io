    Общие темы#

    SaltStack — cистема управления конфигурациями и удалённого выполнения операций.

    • Salt работает по топологии Master(сервер) / Minion(клиент)
    • SaltStack использует для транспорта - ZeroMQ. Позже инженеры из Linkedin добавили возможность работы через raw tcp. Идея заключается в том что Salt Master общается с Salt Minions(далее Мастер и Миньены) через publish/subscribe модель, где мастер публикует задачу, а миньены принимают ее и асинхронно выполняют.
    • Миньоны подключаются к мастеру на порты TCP 4505,4506.
    • Аутентификация миньонов по ключам в Мастере
      • sudo salt-key -L - просмотреть список ключей на мастере
      • sudo salt-key -A - принять ключи полученные с миньона
    • Два формата работы
      • Execution modules - императивны и не идемпотентны. Они написаны как глаголы - скачай архив, удали файл и тд.
      • State modules - декларативны и идемпотентны. Они приводят систему в определенное… состояние описывая финальный результат, а не как к нему прийти - убедись что пакет nginx установлен и он версии 1.21.3 и тп.
    • Исполняемые модули исполняются через команды [[SaltStack#Утилитные команды|Полезные команды]]
        • есть возможность отлаживать код с помощью -l debug в комадах
    • Зерна (GRAINS). Salt ипользует систему, называемую Grains для построения статических данных о Миньонах. Эти данные включают в себя информацию о ОС, CPU архитектуре и много другое

    Salt execution (запуск команд)#

    Состояния (States)#

    • Стейты (state files) — аналог playbooks в Ansible. В них с помощью state.modules, описывается, к какому состоянию нужно привести миньона.
    • Кроме того в SaltStack существует понятие top-файла. Это, по сути, словарь, который помогает удобно группировать миньонов по некоторым атрибутам и указать, какие стейты или роли (если вы пользуетесь ролевой моделью) исполнить.

    Расширенные темы#

    • Pillar - Pillar is an interface for Salt designed to offer global values that can be distributed to minions. Могут хранить чувствительную информацию, распространяются с мастера на миньоны.
    • Mine. The Salt Mine is used to collect arbitrary data from Minions and store it on the Master. This data is then made available to all Minions via the salt.modules.mine module.
    • Salt Returners — по-умолчанию результат исполнения на миньонах возвращается к «мастеру». Returner позволяют переопределить этот output
      • Документация,
      • Можно все события передавать внешнему потербителю
      • Добавление returner
    • Reactor. - модуль выступает в качестве «слушателя», который фильтрует тегированные сообщения и инициализирует какие-то действия по этому поводу.
    • Расписание - запускает каждую команду в своем процессе

    Targeting (назначение на миньоны)#

    • Targeting with Grains. Пример - salt -G 'os:Fedora' test.version
    • Назначение на список. Пример - salt -L 'minion1/minion2/minion3' test.ping
    • Matchers
      • Matchers can be grouped together with parentheses to explicitly declare precedence amongst groups. Пример - salt -C "( ms-1 or G@id:ms-3 ) and G@id:ms-3" test.version
      • Аlternative Delimiters. This is done by specifying an alternate delimiter character between the leading matcher character and the @ pattern separator character. Пример - salt -C "J|@foo|bar|^foo:bar$ or J!@gitrepo!https://github.com:example/project.git" test.ping
    • Compound Targeting (-C)*. Multiple target interfaces can be used in conjunction to determine the command targets. These targets can then be combined using and or or statements. Пример - salt -C "G@os:Debian and webser# or E@db .* " test.version
    • Node Group Targeting. cases, It can be convenient to have a predefined group of minions on which to execute commands. This can be accomplished using what are called nodegroups. Nodegroups задаются в конфигурации мастера

    Salt job management#

    • Job IDs. Job IDs are represented as a jid. Представляет собой время запуска задания
    • Job Cache. The job cache is located in the cachedir under the directory named jobs (/var/cache/salt/master/jobs), хранится Keep_jobs: 24
    • основные команды jobs.list_jobs, jobs.active, jobs.lookup_jid (см список команд)

    Events#

    1. The event system sends two pieces of information. The tag and the data items.
    • The tag is a “/” separated string representing a simple data structure
    • The data will be serialized by Salt into a MessagePack string.
    1. События генерируются через event.send и event.fire_master
      • salt-call event.send 'mycustom/app/tag' '{"app": "osmax"}' with_grains=True
      • salt-call event.fire_master '{"app": "console" }' 'mycustom/appFM/tag'

    Jinja (Рендереры)#

    Общая документация для Salt Документация на шаблонизатор

    • {% … %} - Define a Jinja statement. 
       {% set zone_path = '/etc/named/zones' %}
 {% set users = ['fred', 'bob', 'frank']%}
 {% set items = {
	   'one': {'uid': 9001, 'shell': '/bin/zsh'},
	   'two': {'uid': 9002, 'shell': '/bin/sh'}} %}
   {% set settings = pillar.get(
	   'power_settings', ['standby', 'hibernate']) %}
    • {{ … }} - Print a Jinja expression or call a Salt execution directly at the desired location of the file
    • {# … #} - Jinja comments - not included in output after being rendered Чтение данных
    • grains: all grains for the minion (grains.os_family, grains['os_family'])
    • pillar: all pillar data available to the minion
    • salt: all available execution modules and functions (salt['network.interface_ip']('eth0'))

    for#

    	{% for user in users%}
	create_{{ user }}:
	  user.present:
	    - name: {{ user }}
	{% endfor %}

    Команды работы с состояниями#

    • sudo salt sccm-1 state.sls MODULE_NAME - установка состояние из sls файла
    • salt \* state.highstate - use to execute the state.highstate function
    • salt '*' state.show_sls <SLS_FILE_NAME> - проверить sls файл
    • salt '*' state.apply test - To validate the Salt Formula, run in the test-only (dry-run) mode
    • salt '*' state.apply salt://jinja_pillar](salt://jinja_pillar_sample.sls pillar='{"power_modes": ["croco", "dile"]}' - запуск конкретной формулы и задание pillar
    • salt '*' state.show_lowstate - to see what the low state looks like for a highstate

    Команды работы с grains#

    • salt '*' grains.ls - список доступных grains
    • salt '*' grains.items - получить grains
    • salt -G 'os: CentOS' test.version - проверить выборку по grains
    • salt -G 'puarch: ×86_64' grains. item num_cus - получить конкретный элемент grains

    Команды работы с pillar#

    • salt '*' state.apply pillar='{"key": "value"}' - pillar data can be set at the command line. Не смог заставить работать
    • sudo salt \* saltutil.refresh_pillar - обновить pillar
    • sudo salt-call saltutil.refresh_pillar - синхронизация pillar

    Команды работы с scheduler#

    • sudo salt 'r846695-mi-1.r846695.terra.inno.tech' schedule.show_next_fire_time "xxxx" - как узнать время след.запуска расписания некоторой задачи на миньоне
    • sudo salt 'r846695-mi-1.r846695.terra.inno.tech' schedule.list - список задач для выполнения
    • sudo salt 'r881721-mi-1*' schedule.job_status "highstate" - показать статус запусков по highstate
    • salt-run jobs.active - просмотреть активные задачи
    • ⁠⁠salt-run jobs.list_jobs⁠⁠ - просмотреть ранее выполненные задачи (!! долго исполняется)
    • salt-run jobs.lookup_jid 20190220150112550868 display_progress=True - просмотреть конкретный job даже если он исполняется
    • salt minion-name saltutil.term_job 20190220150112550868 - терминировать job (SIGTERM=15)
    • salt minion-name saltutil.kill_job 20190220150112550868 - удалить job (SIGKILL=9)

    • sudo salt-run state.event pretty=True - вывод очереди

    • salt-run fileserver.dir_list - просмотр директорий модулей
    • salt-run fileserver.file_list - просмотр файлов модулей

    • salt '*' state.show_sls edit.vim --out - отобразить rendered sls
    • salt-call saltutil.running - что сейчас исполняется на миньоне
