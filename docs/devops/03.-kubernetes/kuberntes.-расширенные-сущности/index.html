    Replication Controller#

    ReplicationController предназначен для горизонтального масштабирования и поддержания постоянной работы Pod’а. Если нода отвалится, то ReplicationController создаст реплики на другой ноде.

    • ReplicationController Manifest

    ReplicaSet#

    ReplicaSet — более свежий контроллер репликации, имеет похожий синтаксис, но позволяет использовать более гибкий селектор (разные правила matchLabels, maychExpressions и другие).

    • ReplicaSet Manifest

    Deployment#

    Deployment опирается на ReplicaSet и позволяет объявлять декларативные обновления приложений.

    • Deployment Manifest
    • С Deployment есть нюанс, потому что манифесты всех Pod’ов одинаковые → все Pod’ы в Deployment могут ссылаться на один и тот же PVC, который связан с одним PV → во всех Pod’ах Deployment’а подключён один и тот же PV.
    • Для ограничения количества возможных rollout шагов нужно использовать spec.revisionHistoryLimit
    • Конфигурация Rolling Update

    StatefulSet#

    StatefulSet — ещё одна абстракция в Kubernetes, которая обычно используется для запуска приложений с сохранением состояния (stateful application). Например, базы данных, включая MongoDB и PostgreSQL. StatefulSet гарантирует уникальность имён: при запуске Pod’ов, принадлежащим StatefulSet, каждому Pod’у назначается имя с цифровым индексом (sticky unique identity) от 0 и далее по порядку.

    Использование хранилища в StatefulSet#

    В описании манифеста Pod’а есть специальный раздел PVC-template, в котором описывается темплейт для PVC, и для каждого Pod’а будет создан свой PVC (например, data-test-1), который будет связан со своим уникальным PV.

    DaemonSet#

    DaemonSet гарантирует, что на всех (или нужных) хостах будет запущена копия Pod’а. Когда в кластер добавляется ещё один хост, DaemonSet controller автоматически запускает копию Pod’а на всех хостах кластера. При удалении ноды из кластера DaemonSet удаляет все созданные им Pod’ы.

    • OnDelete. После обновления манифеста DaemonSet новые Pod’ы создадутся только после ручного удаления старых Pod’ов DaemonSet.
    • RollingUpdate (дефолтная). После обновления манифеста старые Pod’ы будут уничтожены, а новые создадутся автоматически. Гарантируется, что в течение всего процесса обновления на каждой ноде будет работать не более одного Pod’а DaemonSet.

    Job#

    Job — сущность в кластере K8s, предназначенная для выполнения задач только один раз. Job поднимает контейнер и выполняет в нём задачу до тех пор, пока Pod не завершится с успешным результатом. Параметры:

    • activeDeadlineSeconds — сколько всего секунд даётся Job на его работу.
    • backoffLimit — количество попыток.
    • Job Manifest Удаление отработавших pod для Job. Pod даже в состоянии Completed продолжает существовать на хосте. При этом, данный Pod не учитывается в списке запущенных на хосте контейнеров, ведь у него статус не Running, поэтому фактически на хосте остаются неиспользуемые Pod. В K8s с версии 1.23 работает автоматическое удаление Job со статусом Complete или Failed через заданный TTL. Настраивается при помощи .spec.ttlSecondsAfterFinished, подробнее о котором можно прочитать в документации.

    Чтобы завершить Job с помощью SIGTERM, если процесс не обратаывает сигнал, то K8s ждёт время, выставленное в terminationGracePeriodSeconds (по умолчанию 30 секунд), и если нет реакции на SIGTERM, посылает SIGKILL. Следовательно, процесс с pid 1 «убивается» в контейнере, и контейнер завершает своё выполнение.

    В pod jobe всегда нужно выставлять restartPolicy: Never, чтобы JobController мог его удалить

    CronJob#

    CronJob создает Job → Job создает Pod → Pod сделал что-то полезное и завершился со статусом Completed

    • CronJob Manifest

    Kubernetes. Deploy и Rollback#

    Vertical Scale#

    • VerticalPodAutoscaler Manifest Варианты updateMode
    • Off — VPA не меняет ничего, только записывает рекомендации в статус объекта VerticalPodAutoscaler.
    • Initial — VPA устанавливает запросы на ресурсы только при создании Pod’а и записывает рекомендации.
    • Recreate — VPA устанавливает запросы на ресурсы для новых Pod’ов и обновляет для существующих, путём их пересоздания. Обновление происходит в тот момент, когда установленные запросы на ресурсы сильно отличаются от рассчитанных рекомендаций.
    • Auto — VPA работает как в режиме Recreate (механизм изменения запросов на ресурсы «на лету» находится в разработке).

    Секция resourcePolicy

    VPA необходимо установит отдельно git clone https://github.com/kubernetes/autoscaler.git && cd autoscaler/vertical-pod-autoscaler/hack && ./vpa-up.sh

    Специальный компонент VPA Recommender отслеживает использование ресурсов Pod’ом и вычисляет рекомендации. Другой компонент VPA Updater проверяет соответствие текущих настроек запросов ресурсов рассчитанным рекомендациям. Рекомендации можно посмотреть через describe

    Horizontal Scale#

    For per-pod resource metrics (like CPU), the controller fetches the metrics from the resource metrics API for each Pod targeted by the HorizontalPodAutoscaler. Then, if a target utilization value is set, the controller calculates the utilization value as a percentage of the equivalent resource request on the containers in each Pod. If a target raw value is set, the raw metric values are used directly. The controller then takes the mean of the utilization or the raw value (depending on the type of target specified) across all targeted Pods, and produces a ratio used to scale the number of desired replicas.

    • HorizontalPodAutoscaler Manifest

    Cluster Autoscaller#

    Если нагрузка на сервисы растёт быстро, и все свободные Node заканчиваются, надо добавлять в кластер новые Node. Чтобы не бросаться делать это вручную, был придуман механизм автомасштабирования кластера — Cluster Autoscaler (CA). Кластер периодически проверяет, не ждёт ли кто-нибудь Node, и, если таковые есть, то увеличивает размер кластера; если есть Node-бездельники — уменьшает размер кластера

    Affinity и Anti-Affinity Теория#

    Node можно тегировать, помечая различными метками, а затем при написании Pod’а указывать nodeSelector Можно указывать предпочтительные affinity

    Affinity разделяется на Node affinity (секция affinity.nodeAffinity) и Inter-pod affinity/anti-affinity (секции affinity.podAffinity и affinity.podAntiAffinity) Типы scheduler

    • requiredDuringSchedulingIgnoredDuringExecution: The scheduler can’t schedule the Pod unless the rule is met.
    • preferredDuringSchedulingIgnoredDuringExecution: The scheduler tries to find a node that meets the rule. If a matching node is not available, the scheduler still schedules the Pod.

    Для сопоставления можено использовать поле operator для задания логического отношения. Типы операторовIn, NotIn, Exists, DoesNotExist, Gt и Lt.
