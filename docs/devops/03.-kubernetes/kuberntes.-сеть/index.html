    Основное#

    • Node Network - сеть, к которой подключены серверы кластера
    • Service Network - сеть длясервисов
    • Pod Network - IP-адреса Pod приложений. Каждая Worker-нода имеет своё выделенное адресное пространство для Pod’ов. Требования
        • Pod’ы на определённой Worker-ноде умели взаимодействовать по сети со всеми Pod’ами на любых Worker-нодах напрямую и без NAT
      • Сервисы на Worker-ноде (такие как kubelet) умели взаимодействовать с Pod’ами на этой Worker-ноде так же напрямую и без NAT CNI или «Container Network Interface» — это спецификация правил сетевого взаимодействия контейнеров, а также правил создания и удаления сетевых интерфейсов для контейнеров. CNI-плагин kubenet — встроенный в kubelet плагин с самой базовой функциональностью.

    Service Network#

    Для работоспособности сервисов созданется dns-service, позволяющий маршрутизировать трафик. Тип сервиса определяет способ доставки трафика до Pod’ов. По умолчанию создаётся сервис типа ClusterIP. Адрес такого сервиса выделяется из диапазона Service Network. На Worker-ноде сервис реализован с помощью технологий балансировки iptables или IP Virtual Server (IPVS) — встроенного в ядро Linux балансировщика для протоколов TCP и UDP. Компонент kube-proxy на каждой Worker-ноде отслеживает события создания сервисов и добавляет соответствующие iptables-правила в NAT-таблицу для трансляции адреса сервиса на адреса Pod’ов или создаёт IPVS балансировщик адреса сервиса на адреса Pod’ов. Другие два типа сервисов NodePort и LoadBalancer решают задачу доступа внешних пользователей к приложениям внутри кластера Kubernetes. Сервис типа NodePort позволяет подключиться к приложению через IP-адрес Worker-ноды. При запуске кластера Kubernetes определяется диапазон портов (30000-32767 по умолчанию), из которого выделяется очередной номер порта для нового сервиса.
    CoreDNS Service NodePort Service
    NodePort Traffic LoadBalancer Traffic
    Cервис типа LoadBalancer создаётся с помощью внешнего балансировщика. При использовании облачных провайдеров такой балансировщик создаётся автоматически при запросе создания сервиса.

    Ingress#

    Основное назначение Ingress состоит в обеспечении внешнего доступа к HTTP/HTTPS-приложениям Пример Ingress Controller - NGINX Ingress Controller, который запускает сам контроллер, Pod’ы с Nginx и создаёт сервис типа LoadBalancer для этих Pod’ов и протоколов HTTP и HTTPS. Контроллер отслеживает события запросов на создание или изменение Ingress-ресурсов и добавляет конфигурацию в Nginx для маршрутизации траффика к приложению. Ingress Manifest

    Kube Proxy#

    Запускаем прокси-сервер Kubernetes: kubectl proxy --port=8080 Теперь можно выполнять навигацию по API Kubernetes Для доступа к этому сервису, используя схему: http://localhost:8080/api/v1/proxy/namespaces//services/<SERVICE-NAME>:<PORT-NAME> Пример : http://localhost:8080/api/v1/proxy/namespaces/default/services/my-internal-service:http/
