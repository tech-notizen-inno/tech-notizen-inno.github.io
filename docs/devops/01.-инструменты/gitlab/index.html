    Downstream pipeline#

    A downstream pipeline is any GitLab CI/CD pipeline triggered by another pipeline. Downstream pipelines run independently and concurrently to the upstream pipeline that triggered them.

    • A parent-child pipeline is a downstream pipeline triggered in the same project as the first pipeline.
    • A multi-project pipeline is a downstream pipeline triggered in a different project than the first pipeline.

    Для запуска используется ключевое слово trigger Управляется запуск с помощью rules или only Можно использовать в дочерном pipeline отличную от конфигурации родительского

    Инструкции gitlab-ci.yml#

    • dependencies - позволяет указать какие артефакты будут использоваться в Job
    • needs - позволяет указать Job, который требуется из предыдущего stage, чтобы начать выполнять данный job (You can ignore stage ordering and run some jobs without waiting for others to complete. Jobs in multiple stages can run concurrently.)
    • tags - позволяет указать метки runners на которых будет запускаться job
    • when: manual - позволяет запустить job вручную, через интерфейс
    • default - You can set global defaults for some keywords. Each default keyword is copied to every job that doesn’t already have it defined. If the job already has a keyword defined, that default is not used.
    • include + rules позволяют определить, когда проводить включение файлов
    • workflow - позволяет указать запускать ли в целом pipeline
    • extends to reuse configuration sections. Входные аргументы - имя job, массив имен jobs
    • hidden jobs.
    • YAML has a feature called ‘anchors’ that you can use to duplicate content across your document. Use anchors to duplicate or inherit properties. Use anchors with hidden jobs to provide templates for your jobs. When there are duplicate keys, the latest included key wins, overriding the other keys.
    • YAML anchors позволяют через выражение !reference добавлять только часть данных из якоря

    Настройка SSH ключей для деплоя#

    1. Сгенерировать с помощью ssh-keygen приватный и публичный ключи
    2. В файловые переменные (file variable) pipeline SSH_PRIVATE_KEY и SSH_PUBLIC_KEY
    3. Зайти в виртуалку для деплоя и добавить публичный ключ в ./.ssh/authorized_keys.
    4. В стадию деплой добавить before_script before_script: # устанавливаем ssh-agent для удобства аутентификации по ssh - 'command -v ssh-agent >/dev/null || ( apt-get update -y && apt-get install openssh-client -y )' - eval $(ssh-agent -s) # сохраняем сгенеренный ранее приватный ключ для раннера - echo "$SSH_PRIVATE_KEY" | tr -d '\r' | ssh-add - # Create the SSH directory and give it the right permissions - mkdir -p ~/.ssh - chmod 600 ~/.ssh - echo "$SSH_KNOWN_HOSTS" >> ~/.ssh/known_hosts - chmod 644 ~/.ssh/known_hosts
    5. Запустить ssh-keyscan host_name.com и внести содержимое в переменную SSH_KNOWN_HOSTS

    Environments#

    Environments describe where code is deployed. GitLab:

    • Provides a full history of deployments to each environment.
    • Tracks your deployments, so you always know what is deployed on your servers. An environment is either static or dynamic:
    • Static environment
      • Usually reused by successive deployments.
      • Has a static name - for example, staging or production.
      • Created manually or as part of a CI/CD pipeline.
    • Dynamic environment
      • Usually created in a CI/CD pipeline and used by only a single deployment, then either stopped or deleted.
      • Has a dynamic name, usually based on the value of a CI/CD variable. Пример статического окружения Пример динамического окружения Пример задания job для выполнения в момент удаления environment
    • Удаление environment вручную

    Yaml справочные материалы#

    Базовый синтаксис

    • отступы из пробелов (символы табуляции не допускаются) используются для обозначения структуры
    • комментарии начинаются с символа (#), могут начинаться в любом месте строки и продолжаются до конца строки
    • списки обозначаются начальным дефисом (-) с одним членом списка на строку, либо члены списка заключаются в квадратные скобки ([ ]) и разделяются запятой и пробелом (, )
    • ассоциативные массивы представлены двоеточием с пробелом (: ) в виде ключ: значение, по одной паре ключ-значение на строку, либо в виде пар, заключённых в фигурные скобки и разделенных запятой и пробелом (, )
    • строки записываются без кавычек, однако могут быть заключены в одиночные или двойные кавычки
    • внутри двойных кавычек могут быть использованы экранированные символы в C-стиле, начинающиеся с обратной косой (****)

    Полезности#

    • Экранирование символов - https://docs.gitlab.com/ee/ci/yaml/script.html
    • чтобы экранировать внутри скрипта символы : (используемые в yml), нужно использовать одинарные кавычки
    • чтобы внутри одинарных кавычек вызвать переменную, нужно ее выделить двойными кавычками
