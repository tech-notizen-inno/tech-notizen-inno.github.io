    Terraform хранит свой конфиг в файлах, написанных на HCL (YAML конфиги). При изменении файлов конфига Terraform автоматически определяет, что уже развернуто, а что следует добавить или удалить. Например, можно изучить инструкцию по развёртыванию через него в Яндекс.Облаке. На самом деле Terraform можно использовать для работы с любым REST API, правда, для API нужен особый плагин под названием провайдер (англ. provider).

    Основные шаги по использованию Terraform#

    1. Определить, из каких элементов состоит инфраструктура.
    2. Написать конфигурационные файлы.
    3. terraform init — установить плагины Terraform, необходимые для используемых элементов.
    4. terraform plan — посмотреть, какие конкретно изменения Terraform внесёт в существующую инфраструктуру.
    5. terraform apply — применить эти изменения.
    6. terraform destroy - удалить конфигурацию Terraform работает с сервисами через провайдеров, поэтому процесс применения изменений можно разделить на несколько шагов:
    7. Определить желаемое состояние ресурса — Terraform считывает переменные из конфигурации и передаёт их провайдеру.
    8. Определить текущее состояние ресурса — Terraform спрашивает провайдера о текущем состоянии ресурса, по сути, Read запрос в API сервиса.
    9. Определить, нужны ли изменения ресурса (создание/удаление/изменение) — Terraform сравнивает значение переменных из первых двух шагов.
    10. Применить изменения, если в шаге 3 утвердительный ответ — Create/Update/Delete запросы в API.

    Основные понятия#

    Провайдер — это плагин для работы с каким-либо сервисом. Для популярных сервисов (облачных и не очень) уже написаны плагины-провайдеры (Шаблон). Ресурс — это описание сущности, которую можно создать в API сервиса (с ним общается провайдер). Datasource — это дополнительный, подключаемый источник данных. Например, с помощью datasource мы можем получить список подсетей в сервисе

    Список команд Terraform#

    • fmt — форматирует файлы Terraform в текущем каталоге
    • init — подгружает всё необходимое для работы в текущем каталоге
    • validate — проверяет файлы Terraform на синтаксические ошибки
    • plan — выводит план изменений
    • apply — применяет изменения конфигурации инфраструктуры
    • destroy — удаляет инфраструктуру, описанную в текущем каталоге
    • output — выводит в терминал значения переменных состояния
    • show — отображает текущее состояние инфраструктуры
    • state — позволяет производить манипуляции с файлами состояния
    • version— выводит текущую версию Terraform
    • graph - построить визуальный вывод в формате GraphViz

    Переменные#

    В Terraform можно параметризовать значения и три типа переменных:

    • Input переменные
    • Output переменные
    • Local переменные

    Input variables#

    variable "instance_zone" {  
	default = "ru-central1-a"  
	type = string
	description = "Instance availability zone"  
	validation { 
	condition = contains(toset(["ru-central1-a", "ru-central1-b", "ru-central1-c"]), var.instance_zone)  
	error_message = "Select availability zone from the list: ru-central1-a, ru-central1-b, ru-central1-c."  
	}  
	sensitive = true  
	nullable = false  
}
    • default — значение по умолчанию, которое затем делает переменную необязательной.
    • type — какой тип должен быть у переменной. Если не указан, то тип может быть любым. Допустимые типы string, number, bool, также есть более «сложные» типы list(<TYPE>), set(<TYPE>), map(<TYPE>) и другие, any — любой тип.
    • description — описание переменной, своеобразная документация.
    • validation — блок для определения правил проверки, обычно в дополнение к ограничениям типа. В примере наложены ограничения на значение переменной zone.
    • sensitive — показывать или нет значение переменной в выводе apply и plan. Подробнее тут. Любые переменные, которые зависят от sensitive, тоже будут sensitive.
    • nullable — определяет, может ли переменная иметь тип null. По умолчанию nullable = true. Если для какой-то переменной установлено nullable в false и определено default, то если Terraform’у передадут значение этой переменной как null, он подставит дефолтное. Обращение к переменной var.<имя переменной>

    Значения переменных#

    Значения переменных можно задать несколькими способами:

    • Через variable или locals.
    • В cli через -var (поштучно): (terraform apply -var="instance_zone=ru-central-c")
    • Через файл с расширением .tfvars (пачками). По умолчанию загружаем значения из terraform.tfvars, но можно явно обозначить файл для загрузки (-var-file="testing.tfvars")
    • Через переменные окружения. Переменная должна начинаться с TF_VAR_, а дальше уже имя переменной (export TF_VAR_instance_zone=ru-central-d)

    Мета-аргументы#

    • depends_on — для указания скрытых зависимостей.
    • count — для создания нескольких экземпляров ресурсов, количество экземпляров = count. Подробности тут.
    • for_each создаёт несколько экземпляров модуля из одного блока модуля. За подробностями сюда. Внутри конфига можно использовать each.key, чтобы получить ключ текущего элемента коллекции, или each.value — для получения текущего значения элемента коллекции.
    • provider — для выбора конфигурации провайдера. Дополнительно читать тут.
    • lifecycle — для настройки жизненного цикла. Сейчас не используется, но зарезервирован на будущее.
