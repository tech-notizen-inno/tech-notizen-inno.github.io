    Как обрабатывается gradle.build.kts#

    Кратко, что происходит «под капотом»:

    • Инициализация

      • Gradle находит корневой проект (settings.gradle/settings.gradle.kts указывает включенные модули).
      • Загружает wrapper-версию Gradle и плагинов.

    • Конфигурация

      • Gradle выполняет Kotlin-скрипт целиком как конфигурационный код:
        • Подключает плагины из блока plugins.
        • Создает/настраивает экстеншены плагинов (например, java, kotlin, jooq, liquibase).
        • Применяет зависимости из repositories/dependencies.
        • Регистрирует и настраивает задачи (tasks.register, tasks.named).
        • Любая логика на уровне верхнего уровня выполняется в фазе конфигурации (например, вычисление версий, чтение свойств, формирование строк подключения).
      • Результат — построенный граф задач с их входами/выходами и зависимостями.

    • Исполнение

      • Gradle выбирает задачи, указанные в командной строке (или задачи по умолчанию), и запускает их в правильном порядке.
      • В этом этапе выполняется «action»-часть задач: компиляция, тесты, запуск контейнеров, миграции, генерация кода и т. д.
      • Инкрементальность: Gradle использует up-to-date checks и build cache; если входы/выходы задачи не изменились, задача может быть пропущена.

    Где выполняется код:

    • Фаза конфигурации — в JVM Gradle (daemon). Любой код на верхнем уровне скрипта и внутри блоков конфигурации (plugins, dependencies, tasks.register { … } конфигурационная часть).
    • Фаза исполнения — actions внутри задач (doLast, task { doLast { … } }, плагины).

    Как использовать переменные/настройки:

    • gradle.properties и системные/ENV переменные можно читать через providers:
      • providers.gradleProperty("dbUrl").orNull
      • providers.environmentVariable("POSTGRES_USER")
    • Для секретов лучше ENV/CI-провайдеры, а не в скрипте.

    Как задаются зависимости задач:

    • Через dependsOn, mustRunAfter/shouldRunAfter, finalizedBy.
    • Плагины часто сами связывают задачи (например, compileKotlin зависит от compileJava и т. п.).

    Как дебажить:

    • ./gradlew tasks — посмотреть задачи.
    • ./gradlew help –scan или –info/–debug — подробные логи.
    • gradle –stop — перезапуск демона, если кэш сломался.
