    Фазы выполнения запроса#

    Анализ и выполнение запроса#

    1. Получение данных
      • seq scan
      • index scan
      • bitmap index scan
    2. Соединение данных (Join)
      • parallel nested loop (внешний набор строк сканируется параллель, внутренний - последовательно)
      • parallel hash join (работает только на равенство)
      • merge join (использует соритровку по табилцам)
    3. Сортировка (order by)
      • quick sort
      • top-N heapsort - когда нужна только часть значений
      • инкрементальная сортировка - данные отсортированы, но не по всем ключам

    • В индекс можно добавлять поля, которые не участвуют в индексе, но это может позволять не читать сами данные, а прочитать только index (строки 2 и 3 таблицы)

    Сравнительный график способов соединения данных

    Для расчета стоимости плана (explain - строит план, explain analyze - строит план, исполняет запрос и сравнивает результаты)

    • собирается статистика (совместно с автоочисткой или analyze)
    • оценка селективности (доля выбираемых строк на основе статистики) и оценка кардинальности (оценочное количество строк набора, вычисляется на основе селективности)
    • оценка стоимости = совокупность оценки каждого из шагов и числа обрабатываемых строк
    • оценка предварительной трудоемкости операции (построение hash таблиц, сортировка и т.д)

    Оценки (costs) плана можно сравнивать только в рамках одного запроса

    Статистика оптимизатора#

    • Команда analyze - сборка статистики по запуску (обычно запускается вместе с автовакуумом)

    Операторы, влияющие на оценку стоимости запроса#

    Переменные, влияющие на оценку стоимости запроса#

    • IO (можно настраивать для табличных пространств) - seq_page_cost = 1.0; (сканирование страниц подряд) - random_page_cost = 4.0; (сканирование произвольной страницы) - effective_io_concurrency = 1.0
    • CPU - cpu_tuple_cost = 0.01; - cpu_index_tuple_cost = 0.005; - cpu_operator_cost = 0.0025; - CREATE FUNCTION … COST (стоимость)
    • Память - work_mem = 4MB; - hash_mem_multiplier = 1; - maintenance_work_mem = 64MB; (выделяеться для серверного процесса) - effective_cache_size = 4GB.
    • Параллельное выполнение: - parallel_setup_cost = 1000; - parallel_tuple_cost = 0.1.
    • Курсоры - cursor_tuple_fraction = 0.1 (отбираемая доля курсора для сбора статистики)

    Уровни настройки параметров#

    • На уровне табличного пространства.
    • На уровне базы данных.
    • На уровне роли и базы данных.
    • На уровне процедуры или функции.
    • На уровне сеанса или транзакции.

    Расширенная статистика#

    1. Создается с помощью create statistics трех разных типов - ndistinct, which enables n-distinct statistics - dependencies, which enables functional dependency statistics - mcv which enables most-common values lists.
    2. Вызвается analyze <object>
    3. После этого план будет уточнен

    Расширения для работы с планами#

    • SR_PLAN, для сохранения рассчитанного плана и принудительного использования его в дальнейшем.
    • Adaptive Query Optimizer, которое реализует обратную связь планировщику от реального времени выполнения запроса, то есть планировщик измеряет фактические результаты выполненного запроса и корректирует свои планы в будущем с учетом этого.
    • pg-wait sampling/p_ sentinet - выборка истории активных сессий.
    • pg_profile - сбор исторических статистик для анализа тенденций.
    • pg_qualstats - сбор статистик по предикатам, консультант по индексам.
    • pg_stat_plans - профиль с точностью до плана, а не запроса.
    • pg_stat_kcache - профиль по CPU и времени на filesystem.
